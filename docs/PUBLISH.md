# GAME100 公開・更新手順（あなたの作業用メモ）

## 1. はじめての公開

### 1-1. GitHub でリポジトリを作る

1. https://github.com/new を開く（ユーザー: `utausnskareshi` でログイン）
2. **Repository name**: `game100`
3. **Public** を選ぶ（GitHub Pages を無料で使うため）
4. 「Add a README file」などのチェックは **すべて外したまま** Create repository

### 1-2. きれいな1コミットで公開する

開発中の細かい履歴は GitHub には載せず、**公開用の1コミット**として送ります
（手元（ローカル）の開発履歴はそのまま残ります）。PowerShell で、プロジェクトのフォルダ
（`game100`）に移動してから実行します。

```powershell
# 開発履歴を含まない「公開用ブランチ」を作り、いまのファイル一式を1コミットにまとめる
git checkout --orphan github-main
git add -A
git commit -m "GAME100 v0.5.0"

# GitHub のリポジトリとつないで送る（公開用ブランチを main という名前で送る）
git remote add origin https://github.com/utausnskareshi/game100.git
git push -u origin github-main:main
```

> 💡 これで GitHub には「1コミットだけ・作者はあなた（`utausnskareshi`）だけ」の
> きれいな状態で公開されます（メールは非公開用 `utausnskareshi@users.noreply.github.com`・
> 個人のメールアドレスは公開されません）。手元の開発履歴（`main` ブランチ）は消えず別で残ります。
> **以降このリポジトリでは、公開用の `github-main` ブランチで作業して `git push` します。**
> さらに確実にするなら、GitHub の Settings → Emails で
> 「Keep my email addresses private」と「Block command line pushes that expose my email」を
> ONにしておくのがおすすめです。

### 1-3. GitHub Pages を有効にする

1. リポジトリの **Settings → Pages** を開く
2. 「Build and deployment」の **Source** を **GitHub Actions** にする

### 1-4. 公開を確認する

1. リポジトリの **Actions** タブで「Deploy to GitHub Pages」が緑✅になるのを待つ（2〜3分）
2. https://utausnskareshi.github.io/game100/ を開く
3. スマホで開くとインストール手順つきのページが表示される

## 2. 更新のリリース（ゲーム追加など）

1. コードを変更する（ゲーム追加は `docs/ADDING_A_GAME.md` 参照）
2. **バージョンを上げる**: `package.json` の `"version"` を変更（**真ん中の数字＝収録した本番ゲームの数**。ゲームを1本ふやすごとに +0.1.0。例: 5本→6本なら `0.5.0` → `0.6.0`・100本で `1.0.0`）
3. **更新履歴を書く**: `src/shell/changelog.ts` の先頭にエントリを追加
   （新ゲームの `addedIn` はこの新バージョンにする → 一覧に NEW バッジが付く）
4. 更新履歴の文面は、公開前に内容を決める（アプリの「せってい → 更新履歴」に出る文章）
5. 動作確認: `npm run build` がエラーなく通ること
6. コミットしてプッシュ（**公開用の `github-main` ブランチ上で**）:

```powershell
git add -A
git commit -m "v0.6.0: ○○を追加"
git push
```

→ あとは自動でビルド・公開されます。

### 利用者にはどう届く？

- インストール済みの端末では、**アプリを次に開いたとき**に新バージョンを検出し、
  「新しいバージョンがあります → 更新する」のトーストが表示されます
- 公開直後は反映まで数分かかることがあります（GitHub Pages のキャッシュ）

## 3. 公開後の実機チェックリスト

土台リリース（ゲーム0個）の時点で確認できるもの:

- [ ] iPhone: Safari で開く → 共有 → ホーム画面に追加 → アイコンから全画面起動する
- [ ] Android: Chrome で開く → インストール → アイコンから起動する
- [ ] 起動後しばらくして「オフラインでもあそべる準備ができました」が出る
      → 機内モードにして開き直しても動く
- [ ] せってい → バージョン表示が正しい / テーマ切替が効く
- [ ] せってい → バックアップの書き出し・読み込みができる
- [ ] きろく → 10×10 のスタンプ台紙が表示される

ゲームを載せたあと（Phase 1 以降）に確認するもの:

- [ ] 縦固定/横固定ゲームで、向きが違うときに回転案内が出る（iPhone）/ 自動で固定される（Android）
- [ ] センサーゲームの開始時に許可ダイアログが出て、傾きの方向が正しい
- [ ] ゲーム中にホームへ戻る → 復帰すると自動でポーズになっている
- [ ] Android の戻るボタン: ゲーム中=ポーズ / ポーズ中=やめる
- [ ] スコア・メダル・実績・レベルが記録される
- [ ] かたむき対応ゲーム（ブロックくずし・かたむきメイロ等）: 傾き操作の反応が自然（強すぎ/弱すぎならゲイン調整を相談）
- [ ] 任意センサーのゲーム（iPhone）: 許可を「許可しない」にしても、タッチ/ドラッグだけで開始して遊べる
- [ ] かたむきメイロ（横固定）: 縦向きにすると「よこ向きにしてください」案内が出る（iPhone）/ 自動で横に固定される（Android）
- [ ] かたむきメイロ: 傾きの向き（右に傾けたらボールが右へ）が正しい / 「すいへい」ボタンで持ち方をリセットできる（※横持ちの左右・上下が逆なら要相談）

## 4. こまったとき

| 症状 | 対処 |
| --- | --- |
| Actions が赤❌ | Actions タブでログを見る。だいたいは build エラー（ローカルで `npm run build` を再確認） |
| ページが404 | Settings → Pages の Source が「GitHub Actions」になっているか確認 |
| 更新が端末に届かない | アプリを一度完全に閉じて開き直す。それでもだめなら「せってい → 表示の修復」 |
| 表示が壊れた | 「せってい → 表示の修復」（記録データは消えません） |
| iPhoneの実機で動きを調べたい | URLに `?debug` を付けて開くと画面内コンソールが出る |

## 5. 仕組みメモ（さわるときの注意）

- 公開URLのパス `/game100/` は `vite.config.ts` の `base`・manifest の `scope`/`start_url` と
  一致している必要がある。**リポジトリ名を変えるならこの3つも変える**
- `dist/` はビルド成果物（コミットしない）。公開は GitHub Actions が自動で行う
- ビルド合計サイズが 15MB を超えるとデプロイが失敗する設定（オフライン用の全ファイル
  事前キャッシュを軽く保つため）。大きな画像・音声ファイルを入れたくなったら要相談
